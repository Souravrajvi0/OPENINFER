# f3 ‚Äî `oi serve` + `oi ui`: the local control plane

**Status:** Draft / discussion
**Owner:** Sourav
**Last updated:** 2026-07-16
**Builds on:** [f1](./f1.md) (runtime-agnostic package manager) ¬∑ [f2](./f2.md) (truth + reliability spine)

> Living design doc. f1 gave vocabulary, f2 gave truth/reliability. f3 makes the
> models **usable everywhere** (one *reliable* endpoint) and **visible** (one lens),
> with the runtime still invisible underneath.

---

## 1. Reframe

Today `oi` is where you *install and run* models. f3 makes `oi` the thing other
tools *talk to* and the place you *watch* what's happening.

> **oi serve** ‚Äî one canonical local OpenAI-compatible endpoint that is a
> **reliability layer** over the runtime (not a naked passthrough).
> **oi ui** ‚Äî a narrow browser lens over that endpoint: catalog, live monitor,
> doctor, playground.

### Why this matters ‚Äî and the honest objection it has to answer

Every serious local-AI tool (Cursor, Cline, aider, Continue, Gemini CLI, MCP
clients, a future Product 2 agent) integrates by pointing at an **OpenAI-compatible
base URL**. But **Ollama already exposes one** at `11434/v1` ‚Äî so a developer can
point Cursor at Ollama *today*. If `oi serve` were just another passthrough, it
would add nothing.

The reason it exists: **raw Ollama silently breaks agents, and `oi serve` fixes
that.** Concretely (all from the f2 research):

- **Silent context truncation.** Ollama defaults `num_ctx` to **2048 tokens**.
  Agents send huge prompts (whole files + system instructions); Ollama silently
  drops everything past 2048, so the model "ignores half your code" and nobody
  knows why. `oi serve` sets the right context length per model + hardware. *This
  one fix justifies the whole endpoint.*
- **Cold-start hangs / "permanent thinking."** No sane request timeout or
  keep-alive ‚Üí the first agent call stalls. `oi serve` applies both.
- **Backend lock-in.** One `oi` URL survives swapping Ollama ‚Üí llama.cpp/vLLM later
  (f1 ¬ß7) without touching every tool's config.
- **Observability.** `oi ui` / `oi top` to see what's actually happening.

So the pitch is **"the reliable local endpoint,"** not "another OpenAI shim." That
also removes Ollama from the *integration surface* ‚Äî the f1 "abstract Ollama" goal.

This mirrors the **gateway product** (`services/gateway`, a governed `/v1` endpoint)
at single-user scale: `oi serve` (local, free, invisible) ‚Üí gateway (team, governed,
observable). Same shape, different scale.

## 2. Principles

1. **OpenAI-compatible is the lingua franca.** Don't invent a protocol. Integration
   is "change one URL."
2. **A reliability layer, not a passthrough.** The endpoint's job is to make local
   models actually work under real tools ‚Äî correct context length, timeouts,
   keep-alive, honest errors ‚Äî not just forward bytes.
3. **Don't hide which model is running.** Consistent with f2: the served model is
   always visible (response header + `/v1/models` + UI). Default behavior is
   deterministic ‚Äî the client gets exactly the model it named.
4. **Runtime stays invisible.** `serve` goes through the `Runtime` seam (f1 ¬ß7); the
   client never learns it's Ollama.
5. **Localhost is trusted; the LAN is not.** Keyless on `127.0.0.1`; any routable
   bind requires auth.
6. **Protect the near-zero-dep install.** `serve` uses Node's built-in `http` ‚Äî no
   web framework. `ui` is a single self-contained static page, lazy-loaded.
7. **The UI is a lens, not a product.** It complements Open WebUI; it does not try
   to replace it (¬ß6).

## 3. `oi serve` ‚Äî the endpoint

```
oi serve                       # http://127.0.0.1:11435  (11434 is Ollama's)
oi serve --port 8080
oi serve --host 0.0.0.0        # LAN ‚Äî REQUIRES an API key (see ¬ß5)
```

Long-running process. Accepts OpenAI-shaped requests, applies reliability defaults,
forwards to the runtime, streams back OpenAI-shaped responses.

### Routes (OpenAI-compatible)
- `POST /v1/chat/completions` ‚Äî chat, streaming (SSE `data: {‚Ä¶}\n\n`, terminal
  `[DONE]`) and non-streaming. The one everything uses.
- `GET  /v1/models` ‚Äî the installed models (real tags), so tools can list/select.
- `POST /v1/completions` ‚Äî legacy text completion (thin shim; add only if a target
  tool needs it).
- `GET  /health` ‚Äî liveness + which model is warm (feeds `oi ui` / `oi top`).

### The reliability layer (the actual point) ‚Äî applied automatically
- **Correct context length.** Set `num_ctx` from the resolved model + hardware
  budget so big agent prompts aren't silently truncated. Never default to 2048.
- **Sane timeouts + keep-alive.** No cold-start hang, no "permanent thinking";
  keep the model warm between calls.
- **Warm / lifecycle.** Keep the served model loaded; unload under memory pressure.
  Clients never learn `ollama ps` / keep-alive exists.
- **Honest errors.** Map runtime crashes to clear messages (reuse
  `classifyCrash()` ‚Äî OOM vs segfault vs GPU), not raw 500s.

### Model selection (transparent by default)
- **Default: pass the real tag straight through** ‚Äî the client gets exactly the
  model it named. Deterministic, no magic.
- **No model given ‚Üí the active model** (`oi use`).
- The served model is always returned in an `x-oi-resolved-model` header.

### Capability aliases ‚Äî OPTIONAL, opt-in, never default *(demoted ‚Äî see note)*
> Deliberately not the headline. Aliasing `model: "coding"` ‚Üí a resolved model is a
> convenience for **config portability across machines**, but it *hides which model
> is running*, which cuts against f2's "don't hide information." So it is **off by
> default**, enabled only via explicit config/flag, and ‚Äî when on ‚Äî the resolved
> concrete tag is always surfaced (`x-oi-resolved-model` + logged). It may not ship
> in v1 at all; build `serve` on transparent real-tag passthrough first and add
> aliases only if the portability need proves real (¬ß8).

### Under the hood
- All model ops go through the **`Runtime` seam** (f1 ¬ß7). If the seam isn't built,
  `serve` is the forcing function ‚Äî do the seam first so a second backend later
  needs no endpoint changes.
- Streaming: translate the runtime's native stream ‚Üí OpenAI chunk format on the fly
  (we already parse Ollama's stream in `chat.ts`/`ollama.ts`).

## 4. `oi ui` ‚Äî the control-plane lens

```
oi ui            # starts serve (if not running) + opens http://127.0.0.1:11435/
```

Served by the **same** `serve` process: `/` ‚Üí the UI, `/v1/*` ‚Üí the API, `/api/*`
‚Üí small internal JSON (status, catalog, running models). One port, one process.

### Scope ‚Äî deliberately narrow (the things terminals do badly)
- **Catalog browse** ‚Äî installed vs available, filters (task, size, fits-here,
  `supportsTools`), the honest metadata from f2 Track B.
- **Live monitor (`oi top`)** ‚Äî which model is warm, VRAM/RAM/CPU, active requests,
  tok/s. Catches the "one stuck request blocking everything" case.
- **Doctor view** ‚Äî the f2 `oi doctor` report, rendered (GPU-idle, speed, fixes).
- **Playground** ‚Äî a minimal chat box to try the active model.

### Explicitly NOT
- A full chat power-tool, RAG, multi-user, plugins. **Do not fight Open WebUI**
  (~139k stars owns that space). Instead *complement* it: users point Open WebUI at
  `oi serve`. `oi` owns model management + reliability + observability; Open WebUI
  owns chat UX.

### Build constraints
- **Single self-contained static page** (inline CSS/JS), lazy ‚Äî only fetched/opened
  on `oi ui`, so `npm install -g` stays near-instant and dependency-free.
- Theme-aware (light/dark). Data via `/api/*` (reuse the `--json` shapes from f2).
- Borrow the gateway dashboard's *aesthetic* for family coherence, not its build.

## 5. Security (the one hard rule)

- **`127.0.0.1` ‚Üí keyless.** Local dev stays frictionless.
- **Any non-loopback bind (`--host`) ‚Üí API key required.** Store key(s) in
  `~/.openinference/config.json`; require `Authorization: Bearer <key>`. Refuse to
  bind a routable interface without a key configured (fail closed, with a message
  telling the user how to set one).
- Never log prompts/keys. An open inference server on a LAN is a resource-abuse and
  data-exposure risk ‚Äî non-negotiable.

## 6. Adjacent (later phases, not f3 core)
- **`oi integrate <tool>`** ‚Äî generate + validate the config snippet for Cursor /
  Cline / aider / Continue / Gemini CLI pointing at `oi serve`. Build a couple, not
  twenty.
- **`oi mcp`** ‚Äî expose models as an MCP server with local-safe defaults. Narrower
  than the OpenAI endpoint; after `serve`.

## 7. Phased rollout

| Phase | Scope | Depends on | Status |
|---|---|---|---|
| **0 ‚Äî Runtime seam** | Route all model ops through a `Runtime` iface; Ollama the one impl | ‚Äî | ‚¨ú prereq |
| **1 ‚Äî serve (reliability)** | `/v1/chat/completions` (stream + non), `/v1/models`, `/health`; **correct num_ctx + timeouts + keep-alive + honest errors**; transparent real-tag passthrough | 0 | ‚¨ú |
| **2 ‚Äî oi ui** | single-page lens: catalog ¬∑ monitor ¬∑ doctor ¬∑ playground | 1 | ‚¨ú |
| **3 ‚Äî LAN + auth** | `--host`, API keys, fail-closed | 1 | ‚¨ú |
| **4 ‚Äî integrate / mcp** | config generators + MCP server | 1 | ‚¨ú |
| **5 ‚Äî capability aliases** *(optional)* | opt-in `model:"coding"` resolution, always transparent | 1 | ‚¨ú maybe |

**Build order rationale:** the seam (0) unblocks everything and is cheap now.
serve-as-reliability-layer (1) is the real developer win ‚Äî it makes local models
work under agents instead of silently truncating/hanging. UI (2) makes it visible.
LAN/auth (3) and integrate/mcp (4) are extensions. Capability aliases (5) are an
optional convenience, not the point.

## 8. Open questions
1. **Default port** ‚Äî 11435 proposed (adjacent to Ollama's 11434). Confirm.
2. **`num_ctx` policy** ‚Äî cap at the model's max, or at what the hardware budget can
   hold, whichever is smaller? (Proposal: `min(model max, hardware-safe ctx)`, and
   surface the chosen value so it isn't hidden.)
3. **Warm policy** ‚Äî keep only the last-used model warm, or allow N? Ties to the
   memory budget from `hardware.ts`.
4. **Do capability aliases earn their place at all?** Leaning: ship `serve` without
   them; add only if the cross-machine portability need proves real in practice.
5. **UI auth when served on LAN** ‚Äî gate `/` behind the same key, or keep the UI
   localhost-only even when the API is exposed? (Proposal: UI localhost-only.)

## 9. Out of scope (f3)
- Multi-tenant / team features (that's the gateway product's job).
- Non-Ollama backends (the seam is designed here; a second impl is later ‚Äî f4).
- RAG, document upload, persona/lorebook, plugins (Open WebUI territory).
- Billing/quotas/telemetry.
