# f1 — Reposition `oi` as a package manager for AI models

**Status:** Draft / discussion
**Owner:** Sourav
**Last updated:** 2026-06-28

> This is a living design doc. It captures the *vision and command model* for `oi`,
> not an implementation plan. No code is committed against it yet.

---

## 1. The reframe

Traditional package managers manage **software**. `oi` manages **AI models**.

```
Traditional            oi
-----------            --------------
Package                Model
   ↓                      ↓
Executable             Inference Engine
   ↓                      ↓
Operating System       Runtime (Ollama, llama.cpp, …)
```

The runtime is **the boring part** — the equivalent of the OS under a package
manager. Today's positioning ("a hardware-aware model package manager on top of
Ollama") chains our identity to one backend. The goal:

> Users should think **`oi` manages models** — not "`oi` manages Ollama."
> Ollama is just the current implementation.

### What this means concretely
- Lead with **models**, not the runtime, in README / banner / help.
- Do **not** delete the runtime from view — **demote** it. Keep it visible in
  `oi info`, `oi status`, and `--verbose`, because the abstraction leaks
  (GPU requirements, model formats, crashes are runtime-specific). Apple hides
  the kernel; `uname` still exists.

---

## 2. Principles

1. **Behave like apt / brew / npm / cargo.** The closer the command surface is to
   what developers already know, the less there is to learn.
2. **Two kinds of state, always visible:** *Installed* vs *Available* (repository).
   This is the core mental model of every package manager.
3. **Runtime-agnostic by design.** Commands describe intent ("install this model"),
   not the engine. Backends are swappable.
4. **No silent heavy actions.** Multi-GB downloads / re-downloads are always explicit.
5. **Backwards compatible.** Renames ship with old names kept as aliases; no scripts break.

## 3. Command surface

Primary names follow package-manager convention. Current names become aliases.

| New (primary) | Alias(es) | Meaning | Status |
|---|---|---|---|
| `oi search <q>` | `find` | Search the catalog (repository) for models | new |
| `oi install <model>` | `pull`, `add` | Download a model | rename of `pull` |
| `oi use <model>` | — | Make a model the active one (installs if missing) | exists |
| `oi list` | `models`, `ls` | Show **installed** models | rename of `models` |
| `oi remove <model>` | `rm`, `uninstall` | Delete a model, free disk | exists as `rm` |
| `oi info <model>` | `show` | Full model detail (see §5) | new |
| `oi update` | — | Refresh the **repository index** (the catalog) | new — see §6 |
| `oi upgrade [model]` | — | Upgrade installed model(s) to newer upstream version | new — see §6 |
| `oi outdated` | — | List installed models with updates available | new — see §6 |
| `oi status` | — | Active model + runtime + hardware | exists |

Notes:
- `search` and `list` both render the **Installed / Available** badge (§4).
- Keep aliases permanent, not deprecation stubs — `rm`/`pull`/`models` are good short forms.

## 4. Installed vs Available (the repository model) — **highest priority**

The single most valuable missing concept. Package managers always show both states.

```
$ oi search vision

  Gemma Vision     not installed     ~6 GB   image
  Qwen2.5-VL       ✓ installed       ~5 GB   image, active
  MiniCPM-V        ✓ installed       ~4 GB   image
  Llama 3.2 Vision not installed     ~8 GB   image
```

### We already have both halves
- **Repository index** = the catalog (`data/models.json`).
- **Installed set** = `ollama list` (via `listModelTags()`).

The work is to **merge and label** them, not to build new data sources. Mark the
active model distinctly. This applies to `search`, `list`, and the wizard picker.

---

## 5. `oi info <model>`

Show everything a user needs to decide, package-manager style:

```
RAM needed          ~5 GB
VRAM (full offload) ~6 GB
Context length      32k
License             Apache-2.0
Quantization        Q4_K_M (default), Q8_0, FP16
Best for            coding, reasoning
Benchmarks          (HumanEval, MMLU, …)
Installed?          ✓ yes — active
Runtime             Ollama (powered by)
```

**Dependency:** most of this metadata is **not in `models.json` today** (license,
context length, quant options, benchmarks). `oi info` is gated on a
**catalog-enrichment** task. Build order: enrich catalog → then `info`.

---

## 6. `update` vs `upgrade` — an architectural fork

Package managers separate two operations we currently conflate:

| apt | Meaning | `oi` today |
|---|---|---|
| `apt update` | Refresh the **repository index** | only via `npm update -g` (catalog is bundled) |
| `apt upgrade` | Upgrade **installed packages** | does not exist |

### Decision needed (blocks the design)
**Should the catalog be independently fetchable, or stay bundled in the npm package?**

- **Bundled (today):** new models require a CLI release. Simple, offline-friendly,
  but conflates "new tool" with "new catalog."
- **Fetchable (`oi update` pulls `models.json` from a URL):** the index updates
  without a tool release — true package-manager behavior. Adds network dependency
  + hosting + versioning concerns.

Recommendation: **lean fetchable**, with the bundled catalog as the offline fallback
(ship a baseline, refresh over network when available). Revisit before building `update`.

### `upgrade` / `outdated` caveat
Ollama tags are **mutable** — `qwen3:8b` can be re-pushed upstream under the same
tag. "Newer version" therefore means **comparing local digests against the registry**,
not comparing semver. Different mechanism than `npm outdated`. Stage this last and
make re-downloads explicit (never a silent multi-GB re-pull).

## 7. Runtime abstraction (future)

Define a `Runtime` interface so the engine is swappable:

```
interface Runtime {
  isAvailable(): boolean
  install(): Promise<void>         // install the engine itself
  pull(model): Promise<void>
  list(): Promise<InstalledModel[]>
  run(model, messages): AsyncIterable<token>
  delete(model): Promise<void>
}
```

Candidate backends: **Ollama** (first impl), llama.cpp, LM Studio, vLLM, TensorRT-LLM.

Honest caveat: these are **not interchangeable**. vLLM/TensorRT are GPU/server-class;
LM Studio is GUI-centric; model file formats differ. The abstraction will leak —
plan for capability flags (e.g. `requiresGpu`, `supportedFormats`) rather than
pretending all runtimes are equal. **Design the seam now; implement the second
backend later.** Nothing else in this spec depends on it.

## 8. Phased rollout

| Phase | Scope | Effort | Value | Status |
|---|---|---|---|---|
| **1 — Identity + vocabulary** | Rename commands (+aliases); reword README/banner away from Ollama-forward; add **Installed/Available** to `search` | low | high | ✅ done (v1.4.0) |
| **2 — `oi info` + catalog enrichment** | `info` shipped with current fields; license/context/quant/benchmarks still need catalog enrichment | medium | high | 🟡 partial — `info` done, enrichment pending |
| **3 — `update`/`upgrade`/`outdated`** | Decide bundled-vs-fetchable catalog; registry digest comparison | medium-high | medium | ⛔ blocked on §9.1 |
| **4 — Runtime abstraction** | Define `Runtime` interface; Ollama as first impl; capability flags | high | strategic | ⬜ not started |

### Shipped in v1.4.0 (Phase 1 + `info`)
- Commands renamed to package-manager vocabulary, old names kept as aliases:
  `install` (`pull`/`add`), `list` (`models`/`ls`), `remove` (`rm`/`uninstall`).
- New `search [query]` (`find`) — searches the catalog, shows installed vs available,
  flags models too big for the machine, marks the active model.
- New `info <model>` (`show`) — RAM, size, quality, best-for, fit, installed state,
  runtime; notes that license/context/quant/benchmarks are coming.
- Shell slash-commands mirror all of the above (`/search`, `/install`, `/info`,
  `/list`, `/remove`) with tab-completion (catalog for install/use/info, installed
  set for remove).
- README + `package.json` description repositioned: "package manager for local AI
  models," runtime demoted to "powered by."

**Key insight:** Phases 1–2 are mostly *naming, wording, and merging two lists we
already have*. They deliver ~80% of the "feels like a real package manager"
perception for ~20% of the effort. Phase 4 is the expensive part and blocks nothing.

## 9. Open questions / decisions needed

1. **Catalog: bundled or fetchable?** (§6) — blocks `update`.
2. **How much runtime to expose?** Proposal: hidden by default, shown in `info`/`status`/`--verbose`.
3. **Where does enrichment metadata come from?** (license, benchmarks) — manual curation vs scraped vs upstream model cards.
4. **`upgrade` UX** — opt-in per model, or `oi upgrade` for all? Confirmation before re-download?
5. **Naming final call:** `remove` vs `uninstall` as the primary (both as aliases either way).
