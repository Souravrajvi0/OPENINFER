# f2 — `oi` as a truth-and-reliability layer (lead: `oi doctor`)

**Status:** Draft / in progress
**Owner:** Sourav
**Last updated:** 2026-07-16
**Builds on:** [f1](./f1.md) (package-manager repositioning)

> Living design doc. f1 gave `oi` package-manager *vocabulary*. f2 gives it a
> *spine*: surface the truth that runtimes hide, and remove the work of acting on
> it. Implementation happens against this doc, starting with `oi doctor`.

---

## 1. Positioning

> **oi is the easiest way to discover, install, and reliably use open-source AI
> models on your machine, regardless of the runtime underneath.**

### The philosophy (the important part)

**Don't hide information. Hide complexity.**

A subtle but load-bearing distinction. Developers *want* to know they're running
Qwen3 14B at Q4. They do **not** want to spend an hour figuring out which quant to
pick, why it's pinned to the CPU, why it OOMs after ten messages, or why Cursor
can't connect. That hour is the complexity `oi` removes — while showing the facts,
not concealing them.

Every f2 feature is one move: **surface a hidden truth + remove the work of acting
on it.**

| Command | Truth surfaced | Complexity removed |
|---|---|---|
| `oi doctor` | "Your RTX 4070 is idle — the model is on CPU" | the exact fix, ready to run |
| `oi info deepseek-r1` | "This is a distilled variant, not the original R1" | no more silent mislabeling |
| `oi install qwen3:14b` | "~3 tok/s here; Qwen3 8B does ~18 with near-identical coding quality" | one keystroke to the better pick |

This reframes the product from f1's "package manager for models" into a **truth +
reliability layer**. The install/use surface is unchanged; the spine becomes
**metadata truth · hardware truth · performance truth**.

## 2. Why the market gap is real (research, July 2026)

Consistent across current guides and troubleshooting write-ups:

- The dominant failures are **"installed but broken,"** not "can't find a model":
  OOM (esp. **KV cache growing over a long conversation**), **silent CPU fallback**
  ("the #1 problem: model loads but runs on CPU at 2–8 tok/s instead of GPU"),
  wrong port / model tag / WSL localhost mismatch. *"Small config mistakes are more
  common than model failures."*
- Choice paralysis: the library is *"overwhelming — names, parameter counts,
  quantization tags, variants."* Universal advice: start with one general + one
  small model.
- Naming confusion: the DeepSeek-R1 distill-vs-full fiasco.
- Integration is universal: every tool connects via an **OpenAI-compatible base
  URL**. Tool-calling **below ~7B is unreliable**.
- The unowned gap, verbatim: Ollama is *"the easiest way to **start** local LLMs,
  but the worst way to **keep running** them."*

**f2 owns the "keep running" gap.** That is why `oi doctor` leads.

## 3. Scope of f2

Three tracks. Track A ships first; B runs in parallel; C depends on both.

| Track | Feature | Risk | Depends on |

| **B** | Catalog metadata + honest `oi info` | low | catalog schema + data |
| **C** | Speed-aware `oi install` nudge | medium | perf estimator (calibrated by A) |

## 4. Track A — `oi doctor` (build first)

A single command that answers "why is my local AI broken/slow?" Self-contained:
new `doctor.ts` + one command in `index.ts`. Touches none of the working
wizard/chat paths.

### Checks (v1)

1. **Runtime reachable** — `pingOllama()`; if down, is it installed
   (`isOllamaInstalled`)? Offer to start it. Report the resolved URL + port so a
   wrong `OLLAMA_URL` / WSL mismatch is visible.
2. **Version** — `GET /api/version`.
3. **Hardware** — reuse `detectHardware()`: RAM, cores, GPU name + VRAM (via
   `nvidia-smi`), disk free, OS. Flag CPU-only explicitly.
4. **GPU actually in use** *(the headline check)* — if a usable GPU exists but a
   model is loaded on CPU, say so and give the fix. Mechanism (verified against
   Ollama API):
   - `GET /api/ps` → per loaded model returns `size` (total) and `size_vram`.
   - `size_vram == 0` with a GPU present → **fully on CPU** → "Your {gpuName} is
     idle." `size_vram < size` → **partial offload** (spilling to RAM → slow).
     `size_vram == size` → fully on GPU (healthy).
   - If nothing is loaded, load the active model with a tiny generate first, then
     read `/api/ps`.
5. **Smoke test + real numbers** — one `generate` (reuse `verifyModel` shape),
   measure **time-to-first-token** and **tokens/sec**. On failure, route through
   `classifyCrash()` (OOM vs segfault vs GPU) — never a raw stack trace.
6. **Disk headroom** — reuse `fitsDisk`/`detectDiskFreeGb`; warn near-full.

### Output shape

Aligned, scannable, one line per check, with a `✓ / ⚠ / ✕` glyph and an
actionable fix on any non-green line. Example (target):

```
  oi doctor

  ✓ Runtime      Ollama 0.5.x  ·  http://127.0.0.1:11434
  ✓ Hardware     32 GB RAM · 16 cores · RTX 4070 (12 GB VRAM)
  ⚠ GPU usage    Model is running on CPU — your RTX 4070 is idle
                 Fix: <exact env/command>
  ✓ Active model qwen3:8b  ·  loaded
  ⚠ Speed        6 tok/s, TTFT 4.1s — slower than expected for this GPU
  ✓ Disk         418 GB free
```

### Real numbers feed Track C

The smoke test produces **measured tok/s on this exact machine**. Persist it
(alongside the `hardware.json` cache) keyed by model + hardware fingerprint. This
becomes the calibration data that turns Track C's estimates from guesses into
grounded ranges.

### Flags

- `--json` — machine-readable output (see §7; ships with doctor as the first
  consumer, since it's cheap and makes doctor scriptable / bug-report-able).
- `--verbose` — raw probe values (per f1 §9.2: runtime detail hidden by default,
  shown here).

## 5. Track B — catalog metadata + honest `oi info` (parallel)

`oi info` today ends with `"License, context length, quantization… coming soon."`
Replace that with the truth.

### Schema additions to `CatalogModel` (`recommend.ts`)

```
params?: string          // "8B", "14B"
quant?: string           // "Q4_K_M"
variant?: 'distill' | 'base' | 'instruct' | ...
baseModel?: string       // for distills: what it was distilled FROM
supportsTools?: boolean  // tool/function-calling reliability (research: shaky <7B)
contextLength?: number
license?: string
note?: string            // free-text honesty line
```

### Data

Populate `data/models.json` via `scripts/build-catalog.mjs`. **Start narrow** —
the known confusables first (DeepSeek-R1 distills → `variant: 'distill'`,
`note: "Distilled from R1 onto a smaller base — not the original DeepSeek-R1."`).
Do the minimum enrichment the features need; do **not** let this become a
150-model data-entry project that blocks A/C.

### Consumers

- `runInfo` — print the real fields + the distill note.
- `supportsTools` powers later capability/agent steering (warn against sub-7B for
  tool-calling) — the metadata lands now, the steering rides Track C/f3.

## 6. Track C — speed-aware `oi install` (after A + B)

Extend the existing fit-warning in `manage.ts:runPull` (~L341). When the requested
model's estimated tok/s is below a "feels usable" threshold, offer a faster
same-category / similar-quality alternative from the catalog (reuse `recommend.ts`
scoring), confirmed via `askYesNo`.

### The estimator (`perf.ts`) — honest ranges, not fake precision

Precise cross-hardware tok/s is genuinely hard; **bad numbers destroy trust faster
than no numbers.** So:

- v1: coarse estimate by `(param bucket) × (execution mode)` where mode ∈
  {GPU-resident, partial offload, CPU}, derived from model size vs VRAM/RAM budget.
  Present as a **labeled range** ("~3 tok/s — likely too slow"), never a bare
  number implying precision. *(Showing it's an estimate is itself "not hiding
  information.")*
- v2: sharpen using the real measurements Track A persisted.

This ordering is why A leads C: doctor measures reality before we predict it.

---

## 7. Cross-cutting: `--json` output

Developer-native composability (validated: terminal devs pipe everything). Ships
first on `doctor` (enables `oi doctor --json` → bug reports / scripts), then
extends to `list` / `search` / `info`. Not theming — theming is deferred as
low-value.

---

## 8. Explicitly parked (decisions, not omissions)

Off the current north star ("easy + reliable + runtime-agnostic"), belong to a
later **team/reproducibility** story:

- Lockfiles / `oi.json` manifests / `name@version` semver.
- `oi update` / `oi outdated` / `oi upgrade` (f1 §6 — still blocked on
  bundled-vs-fetchable).
- Private / scoped registries, org policies.
- A full chat UI. **Do not fight Open WebUI** (~139k stars, owns chat/RAG/
  multimodal). When the `oi` UI comes (f3), it's a narrow **control-plane lens**
  (catalog browse · live monitor · doctor · quick playground) that *complements*
  Open WebUI by exposing `oi serve` for it to point at.

## 10. Open questions

1. **GPU-fix copy** — the exact remediation for silent CPU fallback differs by
   cause (driver missing, CUDA mismatch, Ollama built without GPU, WSL). How many
   cases does doctor v1 diagnose vs. link out for? *(Proposal: diagnose driver-
   present-but-unused + partial-offload; link out for driver install.)*
2. **Metadata source** — manual curation vs. scrape upstream/HF model cards vs.
   read `/api/ps` `details` (`parameter_size`, `quantization_level`) for installed
   models. *(Proposal: read live details for installed models; curate the catalog
   notes by hand, confusables first.)*
3. **"Feels slow" threshold** — what tok/s triggers the install nudge, and does it
   vary by use case (coding vs chat tolerate different latency)?
